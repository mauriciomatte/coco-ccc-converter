v1.0.60 — 2026-06-10

FujiNet — smarter server access: adaptive timeout + listing progress bar
  • ADAPTIVE RETRANSMIT TIMEOUT (Jacobson/Karels algorithm, same as TCP): the client measures each response's
    RTT and keeps a smoothed RTT + variance per session; timeout = srtt + 4×rttvar (cap 3 s, floor 500 ms).
    It used to be FIXED at 3 s — on a fast server that dropped one packet we wasted a full 3 s; now loss is
    detected in ~500 ms. On a slow server (e.g. coconet.ddns.net, ~1.8 s RTT) the timeout rises by itself and is
    NEVER worse than before (3 s cap). More retries (6) with backoff. DIAGNOSIS: the slowness of some servers is
    the server itself (high RTT + packet loss), not the app — proven with tools/tnfsdldiag.ts (the same client
    lists tnfs.fujinet.online in <1 s and a slow home server in tens of seconds).
  • LISTING PROGRESS BAR: OPENDIRX already returns the total item count, so browsing shows "X/N (%)". The bar
    only appears if the load takes more than ~400 ms (a fast load doesn't flash the bar).

Help — COMPLETE overhaul of every tab's manual
  • Meticulous, tab-by-tab review (DSK, K7, GW, BASIC, OS-9, XRoar — FujiNet was already complete): every
    button, field, toggle, dropdown, slider, keyboard shortcut, modal and behavior is now described in the
    manuals (PT and EN, in parallel), keeping all prior content. Examples of what was detailed: multi-select

v1.0.59 — 2026-06-09
--------------------------------------------------------------------------------
FujiNet — FIX: I/O error reading the mounted disk (LSEEK missing the position)
  • After mounting the right .dsk, the CoCo gave an I/O error on DIR. Cause: at MOUNT we advertise TNFS version
    1.2 (0x0102); per the protocol, for version > 0x0100 the LSEEK (0x25) response MUST include the NEW file
    position (u32 LE) after the status byte. Our LSEEK replied with the status only → the board read 4 bytes
    short, desynced the file position, and every sector read landed at the wrong offset (I/O error). Fixed:
    LSEEK now returns status + position. Validated tools/tnfsdirxtest.ts (32/32: LSEEK returns the position and
    READ after seek is correct).

FujiNet — MUCH FASTER server access (client)
  • Listing: the client now uses OPENDIRX (0x17)/READDIRX (0x18) — many entries per round-trip WITH size and
    type, so no per-file STAT. It used to be ~2N+3 round-trips per folder (1 name per READDIR + 1 STAT per
    file); now ~1–3 round-trips. For a folder with 100 files (150 ms RTT) it drops from ~30 s to < 1 s. Auto
    fallback to the old method (OPENDIR/READDIR+STAT) on servers without OPENDIRX. Validated against the real
    hub tnfs.fujinet.online (/COCO, 11 files, 0.91 s) and on loopback.
  • Download: read block raised 512 → 1024 bytes per READ (half the round-trips; no UDP fragmentation).
  • Note: the board is also fast because it reads the disk ON-DEMAND (only the sectors used); the app downloads
    the whole file so it can edit it — so the download isn't instant, but it's ~2× faster now.

Splash — new image
  • Startup screen updated (amostras/new2splash.png → resources/splash/splash.png, 760×567, transparency kept).

v1.0.58 — 2026-06-09
--------------------------------------------------------------------------------
FujiNet — FIX: the board always mounted the FIRST file in the folder (not the chosen one)
  • Root cause: when you pick a file in its browser, the FujiNet STORES that file's POSITION in the directory
    and, at mount time, does a SEEKDIR to that position. Our server didn't implement TELLDIR (0x15)/SEEKDIR
    (0x16) → the seek failed and the board fell back to position 0 (the first file). That's why, with
    desktop.ini + tutstomb.dsk it mounted desktop.ini; and with 01.dsk + 02.dsk, choosing 02 mounted 01.
    Normal TNFS servers (tnfsd) work because they implement these commands.
  • Implemented TELLDIR (0x15) and SEEKDIR (0x16) in src/main/net/tnfsServer.ts. Also, OPENDIR (0x10) now SORTS
    like OPENDIRX (folders first, alphabetical) so the POSITION is the same whichever command the board uses to
    browse and to mount. Validated tools/tnfsdirxtest.ts (28/28: OPENDIR→SEEKDIR(2)→READDIR returns the right
    file).
  • DIAGNOSTICS: the server now logs every OPENDIR/OPENDIRX, SEEKDIR (position → name) and read OPEN to the
    console (shows EXACTLY which file the board opened/mounted).

FujiNet — WiFi server recent folders: own (visible) dropdown instead of the native one
  • The path field now has a chevron (▾) that opens a clickable LIST of recent paths. (The previous native
    dropdown filtered options by the text already in the field and ended up showing nothing — which made it
    look like recents weren't being saved.)

FujiNet — MAIN FIX: the real board now LISTS the WiFi server's files
  • Root cause: the FujiNet firmware lists directories with the EXTENDED TNFS commands OPENDIRX (0x17) and
    READDIRX (0x18), which our server didn't implement → it replied "invalid command" instantly (the board
    opened the host screen at once, but EMPTY). Our loopback tests used the old commands (OPENDIR/READDIR),

    tnfsd wire format (folders-first sort, wildcard filter, EOF, telldir position, per-entry size/dir flag).
    Validated by speaking the protocol byte-by-byte like the board (tools/tnfsdirxtest.ts, 23/23).

  • The board sometimes auto-selected the Windows desktop.ini instead of the disk. The server now HIDES

    listing AND from access by name (stat/read/write).
  • CONFIGURABLE: a new "Hide files from FujiNet" button (Folder mode) opens a manager that LISTS the built-in

FujiNet — "Which IP to use": recommends the ROUTABLE interface (default route), not WiFi blindly

NEW — "Recover" button (K7 tab): rescues degraded tapes that give NO-SYNC
  • 40+ year-old tapes (and imperfect captures) are often SLOW/stretched: the whole FSK is lengthened, so
    the fixed 600 µs threshold misreads the bits → NO-SYNC even when the data is intact. The new "Recover"
    button auto-CALIBRATES the 1/0 threshold, per segment, instead of assuming a fixed speed.
  • How it works: for EACH tape segment it measures the histogram of wave periods and finds the threshold
    that best separates the two groups (2400 Hz = bit 1 / 1200 Hz = bit 0) via OTSU; it also tries a
    threshold sweep (550–775 µs) with and without an amplitude (hiss) gate, and keeps the setting that
    recovers the MOST valid blocks (checksum OK). It auto-follows the tape's real speed (e.g. ~730 µs
    instead of 600).
  • Toggle in the K7 tab (✨ icon); when on, it ignores the manual Threshold/Amplitude and re-decodes by
    itself. The continuous fallback (tapes with no clear gaps) now sweeps thresholds too.
  • First step of the recovery ROADMAP (R1): unlocks slow/stretched tapes. Planned next: visual diagnostics
    (good/bad block map) and fusing multiple captures ("tape RAID").

FIX — "Revert"/"→ No loader": .WAV now loads with the standard reader
  • The .WAV produced when recovering the pure program (Revert / loader-strip) was built with almost no
    timing and failed at the header on a standard load. It is now generated in REAL TIME (lead-in + long
    leader, via the same engine as "→ WAV"), reading reliably on hardware/emulator. The .CAS stays
    fast-load (compact).

  • Hardware-SAFE approach: the slot is sector-doubled (322,560 = 161,280 ×2). Instead of
    re-doubling blindly, we read the current slot and overwrite ONLY the "even chunks" (the
    256B CoCo sectors that de-double reads), PRESERVING the odd ones byte-for-byte — because a
    test on the REAL image showed the odd halves are NOT identical copies (CF leftovers/2nd
    copy). Only the sectors that actually changed are touched; the rest of the .img is intact.
  • Validated on the real MiniIDE image (507 MB, 337 disks): added file reads back with intact
    data, odd chunks preserved, neighbor disk untouched.
  • Switching disks in a container with pending edits writes the current disk back FIRST (no
    data loss). "Save As" of a file-container produces a standalone .dsk of the current disk.
  • RECOMMENDED to confirm on real hardware (write the .img to a CF and test on the MiniIDE).
